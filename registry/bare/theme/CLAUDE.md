# Presets and theme files

Default Tailwind `@theme` CSS files are shipped as `registry:file` items with an explicit `target` path (e.g. `target: "styles/bare-ui-theme.css"`). This is the correct type for plain CSS files that have no `@/` imports to rewrite. Example registry entry:

```json
{
  "name": "theme",
  "type": "registry:file",
  "files": [
    {
      "path": "registry/bare/theme/default.css",
      "type": "registry:file",
      "target": "styles/bare-ui-theme.css"
    }
  ]
}
```

Theme CSS files live under `registry/bare/theme/`. Consumers install via `shadcn add ng-hai/bare-ui/theme` and then `@import` the file in their `globals.css`. They can replace it with their own `@theme` — the separation between tokens (CSS) and styles (`styles.ts`) is intentional.

## Token contract (Radix Colors, 12-step)

The contract in `default.css` follows [Radix Colors](https://www.radix-ui.com/colors/docs/palette-composition/understanding-the-scale), **not** the shadcn `--primary`/`--secondary`/`--muted` set. Two tiers of 12-step scales — a **base** (`gray` neutral chrome + `accent` brand) and a **status** palette (`danger` / `warning` / `success` / `info`, the feedback colors for components like banner/badge/callout/alert/toast; optional — unused vars cost nothing, and the unstyled bare components reference none of them) — each with a solid ramp (`--{scale}-1`…`-12`) and an alpha ramp (`--{scale}-a1`…`-a12`), in `:root` (light) + `.dark`. The four status scales are generated identically (one `generateRadixColors` run per hue) — danger is just the first of four equal hues, not privileged. Steps carry fixed roles: `1` app bg, `2` subtle bg, `3-5` component bg/hover/active, `6-8` borders/focus, `9` solid (the pure color), `10` solid-hover, `11` low-contrast text, `12` high-contrast text. Specials: `--{scale}-contrast` (legible text on the step-9 solid, for `accent` + every status scale), `--{scale}-surface` (translucent panels), `--background`, `--overlay`. `@theme inline` exposes everything as Tailwind utilities (`bg-accent-9`, `text-gray-11`, `border-gray-6`, `bg-accent-a3`, `text-accent-contrast`, `bg-success-3`, `text-danger-11`). Components stay unstyled (empty slots); the contract is the palette consumers fill in.

The default ships **neutral** (accent == a dark gray). It's generated by the **`theme-generator`** item (`registry:file` → `scripts/`): the build-time CLI `gen-theme.ts` plus a **self-contained vendored copy** of Radix's own custom-palette algorithm `generateRadixColors` (`registry/bare/theme/generate-radix-colors.ts`, sibling, imported `./generate-radix-colors`; MIT from `radix-ui/website`, `@ts-nocheck`, do not edit — re-vendor upstream). Installed straight from the GitHub registry (`npx shadcn@latest add ng-hai/bare-ui/theme-generator`) — **no npm publish, no build** — it lands in `scripts/` (not `lib/`), run with `tsx scripts/gen-theme.ts`. Color deps `@radix-ui/colors` + `colorjs.io` + `bezier-easing` are declared as `devDependencies` (these replaced the former Adobe Leonardo + culori). The engine is framework-agnostic and browser-safe (so it could also be used at runtime), but it's distributed only via the generator — there's a single copy. The generator pins `accent-9` to the brand seed, auto-pairs a tinted gray (Radix's natural pairing), tints the background per brand, supports per-mode (`{light,dark}`) seeds, and prints a WCAG self-check that **warns** (never fails) on sub-AA pairs — Radix optimizes perceptual quality over WCAG-2 ratios, so e.g. the Radix red `danger-9` + white is ~3.9:1 (AA-large).
