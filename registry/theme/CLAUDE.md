# Presets and theme files

Default Tailwind `@theme` CSS files are shipped as `registry:file` items with an explicit `target` path (e.g. `target: "styles/ui-theme.css"`). This is the correct type for plain CSS files that have no `@/` imports to rewrite. Example registry entry:

```json
{
  "name": "theme",
  "type": "registry:file",
  "files": [
    {
      "path": "registry/theme/default.css",
      "type": "registry:file",
      "target": "styles/ui-theme.css"
    }
  ]
}
```

Theme CSS files live under `registry/theme/`. Consumers install via `shadcn add ng-hai/ui/theme` and then `@import` the file in their `globals.css`. They can replace it with their own `@theme` — the separation between tokens (CSS) and styles (`styles.ts`) is intentional.

## Token contract (Radix Colors, 12-step)

The contract in `default.css` follows [Radix Colors](https://www.radix-ui.com/colors/docs/palette-composition/understanding-the-scale), **not** the shadcn `--primary`/`--secondary`/`--muted` set. Two tiers of 12-step scales — a **base** (`gray` neutral chrome + `accent` brand) and a **status** palette (`danger` / `warning` / `success` / `info`, the feedback colors for components like banner/badge/callout/alert/toast; optional — unused vars cost nothing, and the unstyled bare components reference none of them) — each with a solid ramp (`--{scale}-1`…`-12`) and an alpha ramp (`--{scale}-a1`…`-a12`), in `:root` (light) + `.dark`. The four status scales are generated identically (one `generateRadixColors` run per hue) — danger is just the first of four equal hues, not privileged. Steps carry fixed roles: `1` app bg, `2` subtle bg, `3-5` component bg/hover/active, `6-8` borders/focus, `9` solid (the pure color), `10` solid-hover, `11` low-contrast text, `12` high-contrast text. Specials: `--{scale}-contrast` (legible text on the step-9 solid, for `accent` + every status scale), `--{scale}-surface` (translucent panels), `--background`, plus the fixed `--black-a1`…`-a12` / `--white-a1`…`-a12` ramps (Radix `blackA`/`whiteA`) — mode-independent constants declared once in `:root` (no `.dark` copy) for tints/scrims that must not flip with the mode (there is no `--overlay` token — scrims use the black ramp directly, e.g. `bg-black-a6`); `black`/`white` are reserved names in the generator, and `tenants.css` never re-emits them (they layer in from `default.css`). `@theme inline` exposes everything as Tailwind utilities (`bg-accent-9`, `text-gray-11`, `border-gray-6`, `bg-accent-a3`, `text-accent-contrast`, `bg-success-3`, `text-danger-11`). Components stay unstyled (empty slots); the contract is the palette consumers fill in.

Generated themes (below) extend — never replace — this contract with an **accent pool**: every `accents` entry in the generator config becomes a named 12-step scale (`--<name>-*` tokens, `bg-<name>-9` utilities) plus a `[data-accent-color="<name>"]` swap block that re-points all 26 `--accent-*` tokens (incl. `-contrast`/`-surface`) for that subtree. It works because `@theme inline` makes utilities emit `var(--accent-9)` directly (they re-resolve per element), and `data-*` props pass through Base UI — so components need no color API: `<Badge.Root data-accent-color="jade">` re-tints a slot filled once with `accent-*` utilities. Only pool keys are valid attribute values; semantic roles are meaning, not identity, and don't swap. `default.css` itself is unchanged and ships no pool — the attribute is inert until a generated theme loads.

The default ships **neutral** (accent == a dark gray). It's generated by the **`theme-generator`** item (`registry:file` → `scripts/`): the build-time CLI `gen-theme.ts` plus a **self-contained vendored copy** of Radix's own custom-palette algorithm `generateRadixColors` (`registry/theme/generate-radix-colors.ts`, sibling, imported `./generate-radix-colors`; MIT from `radix-ui/website`, `@ts-nocheck`, do not edit — re-vendor upstream). Installed straight from the GitHub registry (`npx shadcn@latest add ng-hai/ui/theme-generator`) — **no npm publish, no build** — it lands in `scripts/` (not `lib/`), run with `tsx scripts/gen-theme.ts`. Color deps `@radix-ui/colors` + `colorjs.io` + `bezier-easing` are declared as `devDependencies` (these replaced the former Adobe Leonardo + culori). Per-theme config (write entries with the `defineTheme` helper — `semantics` aliases autocomplete from your `accents` keys): `{ name, accents, semantics?, gray?, background? }`. `accents` is the pool of named seeds; the **first key is the default accent** — `--accent-*` points at it and its hue drives the auto-paired tinted gray (Radix's natural pairing) + brand-tinted background, both overridable and deliberately singular per theme (every scale is bent to the same page, which keeps alpha ramps honest on tinted backgrounds). `semantics` maps roles to an `accents` key (**alias** — role tokens become mode-independent pointers, zero extra scales; a pool key wins over a same-named CSS color) or a color seed (**private scale** — values only, no swap block, no named utilities). Roles are **opt-in** — there are no default seeds; a theme without `semantics` generates the accent pool only, so a standalone drop-in then ships no status scales (declare `danger` etc. if components use them), while under `tenants.css` an undeclared role falls through to the neutral `default.css` it layers over. danger/warning/success/info are the conventional status names (seed from the Radix step-9 values — red `#e5484d`, amber `#ffc53d`, green `#30a46c`, blue `#0090ff`); custom roles (e.g. `premium`) are allowed. Every seed is `string | { light, dark }`. Emission: a standalone per-theme drop-in (values + role pointers + swap blocks + full `@theme inline`) and one `tenants.css` (`[data-tenant]`-scoped values/pointers; swap blocks scoped per tenant so a name outside the active tenant's pool is a no-op; plus a delta `@theme inline` registering only pool scales + custom roles — the contract tokens come from the neutral `default.css` it layers over). Every scale — pool entries and seeded roles alike — is one `generateRadixColors` run sharing the theme's gray + background, pinning step 9 to its seed in **both** modes (per-mode seeds are how step 9 differs across modes). The engine is framework-agnostic and browser-safe (so it could also be used at runtime), but it's distributed only via the generator — there's a single copy. A printed WCAG self-check **warns** (never fails) on sub-AA pairs — Radix optimizes perceptual quality over WCAG-2 ratios, so e.g. the Radix red `danger-9` + white is ~3.9:1 (AA-large). Generator tests live repo-side in `gen-theme.test.ts` (not shipped — it's not in the registry item's `files`).
